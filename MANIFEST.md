# DSO Manifesto v3.0

> **Determinism First**
>
> *Любое решение, которое можно принять заранее, должно быть принято заранее.*
> *Во время исполнения система не должна думать.*
> *Она должна исполнять заранее проверенный план.*

---

## Введение

Современное программное обеспечение построено вокруг постоянного исполнения.

Циклы. Тики. Опрос. Периодические обновления. Планировщики, принимающие решения в Runtime.

DSO предлагает совершенно другую модель. Главная идея: **любое решение, которое можно принять заранее, должно быть принято заранее.** Во время исполнения система не должна думать — она должна исполнять заранее проверенный план.

### Главная цель

Главная цель DSO — **максимально уменьшить неопределённость Runtime.**

Не FPS. Не RAM. Не CPU. Это лишь следствия.

Главная цель — сделать систему максимально:

- **предсказуемой**
- **проверяемой**
- **объяснимой**
- **детерминированной**

---

## Принципы

### 1. Determinism First

Детерминизм — не одинаковый результат. Детерминизм — это возможность заранее объяснить:

- почему объект существует;
- почему он проснулся;
- кто его разбудил;
- какие ресурсы он использует;
- что произойдёт дальше;
- когда он снова уснёт.

Никакой скрытой магии Runtime.

---

### 2. Compile Knowledge

Runtime — не место принятия решений. Runtime — исполнитель.

Все знания максимально переносятся заранее:

- память
- зависимости
- граф
- события
- ресурсы
- lifetime
- проверки
- маршруты исполнения

---

### 3. Global Planning

Перед запуском система максимально строит план.

Строится:

- **Dependency Graph** — граф зависимостей между объектами
- **Lifetime Graph** — граф жизненных циклов
- **Resource Graph** — граф владения ресурсами
- **Memory Graph** — граф памяти и аллокаций
- **Event Graph** — граф распространения событий

Runtime ничего не вычисляет заново.

---

### 4. Verify Early

Любое действие сначала проверяется. Не существует неверифицированного исполнения.

```
  Event
    │
    ▼
  Verification
    │
    ▼
  Execution
    │
    ▼
  Propagation
```

---

### 5. Local Execution

После построения графа каждый объект исполняется локально. Нет глобального обхода всего мира. Нет бесконечного `Update()` / `Tick()`.

---

### 6. Sleep By Default

Любой объект по умолчанию спит. Активность — исключение. Нормальное состояние системы — сон.

---

### 7. Nothing Executes Without Reason

Никакой объект не имеет права исполняться без причины.

Причины:

- событие
- изменение состояния
- зависимость
- контракт
- таймер
- внешний сигнал

Если причины нет — объект не существует для CPU.

---

### 8. Everything Is An Object

Не существует специальных сущностей. Игрок. NPC. Поезд. LLM. Файл. Процесс. Окно. Документ. База данных. Контроллер.

Все являются одним типом объекта:

```
Object
├── ID
├── Type
├── State
├── Lifetime
├── Resources
├── Contracts
├── Dependencies
├── Actions
├── Events
├── Verification
├── Presentation
├── Priority
└── SleepState
```

---

### 9. World Is A Dependency Graph

Мир — не список объектов. Мир — граф причинно-следственных связей.

```
  Player
    │
    ▼
  Tree
    │
    ▼
  Wood
    │
    ▼
  Factory
    │
    ▼
  Steel
    │
    ▼
  Train
    │
    ▼
  City
```

Изменение распространяется только по графу.

---

### 10. Contracts Instead Of Hidden Logic

Поведение определяется контрактами. Не скрытым кодом.

```
  Factory
  ├── Needs:    Iron, Coal, Workers
  ├── Produces: Steel
  └── Wake:     IronChanged, CoalChanged, PowerChanged
```

Контракт — первичный источник поведения.

---

### 11. Actions Are Objects

Action — не функция. Action — узел графа.

```
  Attack
  ├── Needs:    Weapon, Target
  ├── Produces: Damage, Animation, Sound, Events
  └── ...
```

Любое действие — объект.

---

### 12. Presentation Is Independent

UI. Анимации. Звуки. Эффекты. Они не содержат логики. Они подписаны на события. Логика существует отдельно.

---

### 13. Every Resource Has An Owner

Память. GPU. Файл. Поток. Буфер. Сокет.

Любой ресурс имеет владельца. Никаких анонимных ресурсов.

---

### 14. Total Processing

Любой объект обязан закончить обработку. Даже ошибка — формальное состояние. Нет `Unhandled`. Есть `Fallback`.

---

### 15. Time Is Also An Event

Время — обычный объект. Нет необходимости обновлять мир 60 раз в секунду. Если следующее событие через 5 минут, система может сразу перейти к нему.

---

### 16. One Runtime

OCR. LLM. Speech. DSP. Physics. Gameplay. Database. Operating System.

Не отдельные Runtime. Один универсальный исполнитель.

---

### 17. Universal Object Runtime

DSO — не игровой движок. Не ОС. Не библиотека. DSO — универсальный Runtime объектов.

---

## Что это даёт

### Игры

Вместо `Update()` / `Tick()` получаем:

```
  Wake → Execute → Sleep
```

NPC не обновляются постоянно. Экономика просыпается только когда произошло событие. Игрок не является центром мира. Мир существует самостоятельно.

### DSO World

Главная демонстрация философии. Миллион объектов. При этом активны 20. Все остальные спят. Мир продолжает жить, даже если игроков нет.

### DSO World IR

Любая игра теоретически может экспортировать не код, а смысл объектов:

```
  Object → State → Contracts → Dependencies → Actions → Resources
```

Это позволяет строить адаптеры между разными движками и использовать общую семантику объектов.

### AI

Whisper, LLM, OCR, TTS, DSP, Vision — становятся одним Execution Graph.

### Embedded

ESP32. STM32. PLC. Просыпаются только по событиям. Лишняя работа отсутствует.

### Operating System

Процессы. Файлы. Окна. Драйверы. Планировщик. Всё существует как Dependency Graph.

### Database

Не строки. Не таблицы. А объекты с жизненным циклом, контрактами, зависимостями и событиями.

### CAD

Изменение одного болта не перестраивает всю модель. Активируются только затронутые объекты.

### IDE

Изменился файл. Просыпаются только затронутые: AST, индекс, диагностика, автодополнение.

### DSP

Эффекты. Эквалайзер. Обработка. Работают как граф событий. Не выполняются без причины.

## Итоговая философия

DSO — это не про память. Не про ECS. Не про Arena. Не про игры. И даже не про детерминизм сам по себе.

Это **общая методология построения вычислительных систем**, основанная на пяти фундаментальных идеях:

1. **Determinism First** — любое решение, которое можно принять заранее, должно быть принято заранее.
2. **Compile Globally** — максимально анализировать, планировать и строить графы до исполнения.
3. **Verify Early** — проверять корректность до выполнения действий.
4. **Execute Locally** — выполнять только локально затронутые объекты и зависимости.
5. **Sleep By Default** — любое вычисление должно иметь формальную причину; отсутствие причины означает отсутствие исполнения.

### Ключевая мысль

Сегодня большинство программ отвечает на вопрос: *"Что нужно выполнить сейчас?"*

DSO отвечает на другой вопрос: *"Почему это вообще должно выполняться?"*

Если формальной причины нет, система не должна выполнять работу.

---

> **Важное замечание.** Всё это пока представляет собой архитектурную гипотезу, а не доказанную теорию. Часть принципов (событийная обработка, графы зависимостей, предварительное планирование) уже используется в отдельных областях. Новизна DSO будет определяться тем, удастся ли показать, что единая методология действительно даёт измеримые преимущества сразу в нескольких классах систем (AI Runtime, игровые симуляции, embedded, базы данных, DSP). Именно работающие прототипы и бенчмарки смогут подтвердить или опровергнуть эту гипотезу.
